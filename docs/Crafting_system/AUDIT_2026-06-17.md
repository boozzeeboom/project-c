## 🔴 КРИТИЧЕСКИЕ БАГИ (нужно фиксить ASAP)

### B1. CollectRpc — нет проверки владельца (Anti-grief дыра)

**Файл:** `CraftingServer.cs:327-374`

Любой игрок может вызвать `CollectRpc` и забрать готовый результат чужого крафта. Нет ни `job.OwnerClientId != clientId` проверки, ни проверки на `job.State == CraftingJobState.Completed`.

**Проект:** `10_DESIGN.md` §5.6: «Забирает заказчик»; `00_OVERVIEW.md` §5.6: «Только owner»; `50_KNOWN_ISSUES.md` §3: «Не-owner крадёт — TryCollect/Cancel/Start проверяют ownerClientId».

**Фикс:** Добавить ДО выдачи предметов:
```csharp
if (job == null || job.OwnerClientId != clientId)
{
    SendResultToClient(clientId, CraftingResultDto.Denied(CraftingResultCode.NotOwner, "Только владелец может забрать результат", stationNetId));
    return;
}
if (job.State != CraftingJobState.Completed)
{
    SendResultToClient(clientId, CraftingResultDto.Denied(CraftingResultCode.NotStarted, "Крафт ещё не завершён", stationNetId));
    return;
}
```

### B2. CancelCraft — двойной возврат ресурсов (Duplication bug)

**Файлы:** `CraftingServer.cs:286-323` + `CraftingStation.cs:152-172`

`CancelCraftRpc` возвращает Committed в инвентарь (строки 299-310), а затем `ServerCancelCraft()` копирует Committed→Buffer (строка 161-163). Ресурсы возвращаются И в инвентарь, И остаются в буфере станции как «живые» предметы. Следующий игрок может положить 0 реальных предметов и скрафтить результат.

**Проект:** `00_OVERVIEW.md` §5.2: «Откат: вернуть CommittedItems в инвентарь заказчика».

**Фикс:** Убрать возврат в инвентарь из `CancelCraftRpc` (оставить только `ServerCancelCraft`), и в `ServerCancelCraft` добавить возврат предметов в инвентарь:
```csharp
// В CraftingStation.ServerCancelCraft():
// Перед Clear: вернуть committed → inventory
foreach (var c in job.Committed) { /* AddItemDirect */ }
job.Buffer.Clear();
job.Committed.Clear();
// state = Empty (не Buffered!)
```

## 🔴 СУЩЕСТВЕННЫЕ ПРОПУСКИ ДИЗАЙНА (функциональные дыры)

### D1. StartCraft не валидирует ингредиенты рецепта

**Файл:** `CraftingServer.cs:223-283`

`StartCraftRpc` проверяет только `job.Buffer.Count == 0` (строка 238), но НЕ проверяет, что буфер содержит правильные предметы в правильных количествах для выбранного рецепта (`recipe.Ingredients`). Можно скрафтить рецепт с любым мусором в буфере.

**Проект:** `00_OVERVIEW.md` §5.2: «Проверить, что буфер содержит все RecipeData.ingredients[]»; `10_DESIGN.md` §3.4: `CheckBufferCoversRecipe`.

**Фикс:** Добавить валидацию после получения recipe:
```csharp
if (!ValidateBufferMatchesRecipe(job, recipe, out string reason))
{
    SendResultToClient(clientId, CraftingResultDto.Denied(CraftingResultCode.NotEnoughResources, reason, stationNetId));
    return;
}
```

### D2. MetaRequirement tool check не вызывается

**Файл:** `CraftingStation.cs:115-128` (метод `CanStartCraft`) vs `CraftingServer.cs:223-283` (никогда не вызывает)

`CraftingStation.CanStartCraft` содержит логику MetaReq tool check (строка 123: `_metaRequirement.CanPlayerUse`), но `CraftingServer.StartCraftRpc` НИГДЕ не вызывает `CanStartCraft`. Станции с требованием инструмента (наковальня с молотком) будут работать без инструмента.

**Проект:** `ANALYSIS_REUSE_2026-06-10.md` §3.1: «Tool check через MetaRequirement (как Mining)».

**Фикс:** Добавить в `StartCraftRpc`:
```csharp
var cs = station.GetComponent<CraftingStation>();
if (cs != null && !cs.CanStartCraft(clientId, out string reason))
{
    SendResultToClient(clientId, CraftingResultDto.Denied(CraftingResultCode.MetaReqDenied, reason, stationNetId));
    return;
}
```

### D3. CraftSpeedMultiplier нигде не применяется

**Файл:** `CraftingServer.cs:257` — хардкод `speedMult=1f`

Время крафта всегда равно `recipe.CraftSeconds`, игнорируя `CraftingStationConfig.CraftSpeedMultiplier`. Станция с `craftSpeedMultiplier=2.0` должна крафтить вдвое быстрее.

**Проект:** `10_DESIGN.md` §2.1: поле `craftSpeedMultiplier`; `00_OVERVIEW.md` §5.4.

**Фикс:** 
```csharp
float speedMult = cs?.Config?.CraftSpeedMultiplier ?? 1f;
float duration = Mathf.Max(0.5f, recipe.CraftSeconds / speedMult);
```

## 🟡 ТЕХНИЧЕСКИЙ ДОЛГ (средний приоритет)

### T1. Reflection-вызов CompleteCraft

**Файл:** `CraftingWorld.cs:157-159`

```csharp
var mi = st.GetType().GetMethod("CompleteCraft");
if (mi != null) mi.Invoke(st, null);
```

Это «late-binding forward reference» сделан, когда T-C02 писался до T-C04. Теперь оба класса существуют — нужно сменить `Dictionary<ulong, MonoBehaviour>` на `Dictionary<ulong, CraftingStation>` и вызывать `station.CompleteCraft()` напрямую.

### T2. Двойной реестр ItemId: InventoryWorld vs CraftingWorld

`CraftingWorld` имеет свой `_itemsById` реестр (строки 24-26), а `RecipeData.ResolveItemIds()` использует `InventoryWorld.Instance.GetOrRegisterItemId()`. Два независимых маппинга `ItemData→int` могут разойтись:

- Клиент регистрирует item через `CraftingWorld.RegisterItem()` (из UI, строка 357 `CraftingWindow.cs`)
- Сервер резолвит через `InventoryWorld.GetOrRegisterItemId()` (из `RecipeData.ResolveItemIds()`)

**Рекомендация:** убрать CraftingWorld item registry; всегда использовать `InventoryWorld` для маппинга.

### T3. CraftingWindow вызывает CraftingWorld (server-only static) с клиента

**Файл:** `CraftingWindow.cs:291,304,354,357`

Клиентский UI вызывает `CraftingWorld.RegisterRecipe()`, `CraftingWorld.GetRecipe()`, `CraftingWorld.GetItemId()`, `CraftingWorld.RegisterItem()`. На чистом клиенте (без хоста) `CraftingWorld.IsInitialized == false`, и эти вызовы вернут `-1` / `null`.

**Рекомендация:** добавить client-side кеш в `CraftingClientState` и маршрутизировать UI-запросы через него.

### T4. CancelCraftRpc — нет проверки owner

`CancelCraftRpc` (строка 287) не проверяет `job.OwnerClientId != clientId`. В отличие от критического `CollectRpc`, здесь это менее опасно (нельзя украсть результат, только испортить чужой крафт), но всё равно нарушает дизайн.

### T5. ServerCollect не сбрасывает до Empty на серверной job-модели

`CraftingStation.ServerCollect()` вызывает `job.Reset()` (строка 181), что очищает всё. Но перед этим `CraftingServer.CollectRpc` уже выдал предметы (строка 338-363). Если выдача прошла, а `ServerCollect` по какой-то причине не сработал — предметы задвоятся.

Прогресс отображается в двух местах: окно крафта (UXML ProgressBar) и toast-контроллер (создаётся в коде). Это намеренно, но не очевидно из кода.

`RecipeOutput` в `RecipeData.cs` не содержит `ShipKeyBinding` поля (только `ItemData item`). В дизайне (`10_DESIGN.md` §2.2) было поле `shipKeyBinding`. Выдача корабля через ключ-предмет работает (Item_Key_Ship*), но сам концепт из дизайна не отражён в SO.

### L5. CraftingStation.OnInteracted — заглушка

Строка 206-208: метод ничего не делает кроме лога. Взаимодействие идёт через `NetworkPlayer.TryInteractNearestCraftingStation()`, а не через `IInteractable.TryInteract()`. Два параллельных механизма.

## 🟢 ЧТО СДЕЛАНО ХОРОШО

1. **Архитектура:** чёткое разделение server-authoritative state (CraftingWorld + CraftingJob) и client projection (CraftingClientState + SnapshotDto). Соответствует паттернам GatheringServer/MarketServer.

2. **DTO-дизайн:** `INetworkSerializable` с null-string writeback (P16d). Snapshot содержит server-computed `progress` (избегает clock drift между ServerTime и realtimeSinceStartup — FIX T-C07).

3. **Интеграция:** правильно добавлены RPC-targets в `NetworkPlayer`, авто-спавн в `NetworkManagerController`, регистрация в `InteractableManager`, отдельный `CraftingTimeService`.

4. **UI:** отдельный UIDocument (не tab в MarketWindow), правильно копирует DialogWindow-паттерн (Show/Close, cursor lock, pickingMode). Кнопки `+1`/`+Все` — осознанно вместо drag-and-drop.

5. **Анимация станции:** emission pulse + scale wiggle при `InProgress`, яркая вспышка при `Completed`. Использует MaterialPropertyBlock.

6. **Rate-limit + retry-snapshot:** защита от спама и корректная обработка race condition (NetworkPlayer ещё не заспавнен — retry до 10 кадров).

7. **Документация:** 9 md-файлов, покрывающих архитектуру, дизайн, имплементацию, верификацию, инспектор, known issues, changelog, roadmap и анализ reuse. Исключительно качественно для ~12-15 часов работы.
