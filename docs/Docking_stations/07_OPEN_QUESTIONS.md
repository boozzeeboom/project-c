## B. Главные архитектурные сдвиги после решений

### B1. Сервер — single source of truth (Q3)

**Было:** «Session-only, без persistence».
**Стало:** Сервер держит `_occupiedPads : Dictionary<padKey, occupant>` в
`DockingWorld`. Клиент **не хранит** своё представление — запрашивает
через RPC или получает push при изменениях. Архитектура уже совместима
с NPC-кораблями (Phase 2): `_occupiedPads[padKey] = clientId` работает
одинаково для игрока и NPC.

**Влияние на файлы:** `02_V2_ARCHITECTURE.md` §6 (DockingWorld API уже
правильный), `05_FLOW_AND_INTERACTION.md` §3.2 (где было «session-only»).

### B2. Без хардкода pads (Q4)

**Было:** Default layout = 6 pads (3L+M, 2M+H, 1H).
**Стало:** `DockPadLayout` SO принимает **любое количество** pads.
Ограничение ≤10 на класс для MVP — это **soft-limit** в `OnValidate` SO,
не в коде сервера. Дизайнер сам расставляет.

**Влияние на файлы:** `02_V2_ARCHITECTURE.md` §2.2 (DockPadLayout — без
дефолтного списка), `03_ZONES_AND_TRIGGERS.md` §6 (маппинг pad ↔ shipClass),
`06_ROADMAP.md` T-DOCK-11 (создаём только **пустой** DefaultDockPadLayout,
дизайнер наполняет).

### B3. Двусторонняя связь (Q7)

**Было:** Только кнопки действий, нет «текстового» ответа игрока.
**Стало:** Игрок в CommPanel имеет:
- `Запросить посадку` → ожидание → ответ диспетчера (pad #N) → игрок
  подтверждает: **`Хорошо`** / **`Отбой`**. Это второй round-trip.

**Влияние на файлы:** `02_V2_ARCHITECTURE.md` §5 (новый RPC
`RequestConfirmAssignmentRpc`), `04_DIALOG_AND_DISPATCHER_UI.md` §3
(новый state `Assigned` имеет два под-state: `AwaitingConfirm` и `Confirmed`).

### B4. Departure — отдельная подсистема (Q8)

**Было:** F = boarding (безусловно).
**Стало:** F = boarding безусловно (как было). **НО** при вылете из
OuterCommZone игрок должен сначала через T → `Запросить вылет` → ожидание
→ разрешение. Без разрешения = **toast-предупреждение** (не блокировка).

**Это НЕ часть docking.** Departure = отдельный хаб `DepartureServer` +
`DepartureClientState` + `DeparturePanelWindow` (или встроен в CommPanel).
**Новый раздел** в roadmap: `T-DEPART-00..05`.

**Влияние на файлы:** `05_FLOW_AND_INTERACTION.md` §2.6 (новый сценарий),
`06_ROADMAP.md` (новые тикеты).

### B5. F внутри CommPanel = boarding (Q9)

**Было:** F игнорируется внутри CommPanel.
**Стало:** F внутри CommPanel = стандартное boarding/выход из кресла.
CommPanel **закрывается** (как Esc). Это не две конфликтующих команды —
игрок явно выбирает «закрыть и сесть/выйти».

**Влияние на файлы:** `05 §1.4` (F-key pipeline — CommPanel не перехватывает F).

### B6. T вне кресла игнорируется (Q10)

**Было:** Кнопка disabled + tooltip.
**Стало:** T **никак не реагирует** вне кресла (нет UI, нет toast).
Это упрощает код: `TryInteractNearestDockStation` сразу проверяет
`localPlayer.IsInShip && localPlayer.IsPilot` (новое условие).

**Влияние на файлы:** `05 §1.4` (T handler добавить check).

### B7. Цифры на mesh'е (Q13)

**Было:** Нет labels, только в CommPanel.
**Стало:** Каждый pad в сцене имеет визуальную метку (текстура/материал
с цифрой `5`, `6` и т.п.). Расставляет дизайнер (или ставится автоматически
из `DockPadLayout.padId` через Editor tool в Phase 3). MVP: дизайнер
сам рисует цифры в ProBuilder/Blender.

**Влияние на файлы:** `03 §5.5` (Pad Model — без изменения кода, только заметка
для дизайнера), `06 T-DOCK-12` (в чеклисте расстановки).

## C. Новые требования (после решений)

### C1. `IsPilot` флаг

Нужен флаг «игрок в кресле пилота» (а не просто `IsInShip`). Сейчас
`NetworkPlayer.IsInShip` — игрок сидит в корабле, но может стоять
на палубе. Для T-key нам важно: **игрок именно в кресле пилота**.

**Решение:** использовать `NetworkPlayer.CurrentShip.IsPilot(playerId)` —
новая логика, или проверить `PilotSeatController.IsPilotOccupant`.
**Тикет:** `T-DOCK-14` (вне scope MVP, но нужно для T-key check).

**Workaround для MVP:** T реагирует если `IsInShip` (грубая проверка).
В фазе 2 — точная проверка через `PilotSeat`.

### C2. `RequestConfirmAssignmentRpc` (новый RPC)

После того как сервер назначил pad и отправил `DockingAssignmentDto`,
клиент должен **подтвердить** («Хорошо» = принять, «Отбой» = отклонить).
Сервер только после подтверждения помечает pad как `_occupiedPads[padKey] = clientId`.

**Зачем:** даёт игроку шанс отказаться, не блокируя pad.

**Тикет:** `T-DOCK-01b` (дополнение к T-DOCK-01).

### C3. `DepartureServer` — новая подсистема

**Подробности в `08_DEPARTURE_SUBSYSTEM.md`** (новый файл). Q8 вводит
отдельный хаб для запросов вылета. Архитектурно похож на docking, но
**другая семантика** (не «назначить место», а «получить разрешение»).

**Решение:** это в **отдельном каталоге** `docs/Departure/`, не в этом.
Roadmap в `06` помечает `T-DEPART-*` как **out of MVP scope** (Phase 1.5
после docking MVP).

### C4. UI для подтверждения в CommPanel (Q7)

`CommPanelWindow` добавляет **два под-state** в `Assigned`:

| Под-state | Кнопки | Что показывает |
|-----------|--------|----------------|
| `Assigned` (сразу после назначения) | `[Хорошо] [Отбой]` | «Диспетчер: назначаю pad #N. Подтверждаете?» |
| `AwaitingTakeoff` (только в фазе 1.5) | `[Ждать]` | (после T для вылета) |

Для MVP только первый под-state. Второй — после T-DEPART-*.
