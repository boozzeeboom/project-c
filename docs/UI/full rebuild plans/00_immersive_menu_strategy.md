# 00 — Immersive Menu Strategy (вместо Vista-Style)

> **Тип документа:** Стратегический аналитический разбор. Не план кодинга.
> **Дата:** 2026-07-03 · **Автор:** Mavis (аналитическая сессия)
> **Контекст:** Текущие окна UI Toolkit работают технически (см. `docs/UI/UI_TOOLKIT_GUIDE.md`), но визуально и поведенчески соответствуют **стандартному desktop-окну**: статичная панель, рамка, табличные данные, мгновенный show/hide. NMS/Breath of the Wild/Hades — другой жанр: меню **встроено в экосистему**, а не «слой поверх».

---

## TL;DR — почему сейчас «vista-style»

В проекте есть **вся инфраструктура**, чтобы делать immersive-меню, но фактически реализован только «рабочий слой»:

| Есть ✅ | Нет ❌ (и это и есть «vista-style») |
|---|---|
| UI Toolkit, централизованные окна (`UIManager.OpenPanel/ClosePanel`) | Анимации открытия/закрытия (`schedule.Execute(50ms)` — это костыль MarkDirtyRepaint, не анимация) |
| `UITheme` (51+ цветов, ScriptableObject) | Motion design (timing/easing/curves) |
| Приоритеты и z-ordering | Звуковой дизайн UI (поля `ClickSound/OpenSound/CloseSound` объявлены, но не наполнены и не вызываются на UX-событиях) |
| CharacterWindow (5+ табов), QuestTracker (HUD), DialogWindow (typewriter), InventoryWheel (GL) | **Согласованный стиль переходов между окнами** — каждое окно сейчас существует отдельно |
| Дневной/ночной цикл (`Assets/_Project/Volumes/DayNight`) | Связь UI с состоянием мира (UI не реагирует на смену дня/ночи, погоду, состояние игрока) |
| GDD-14: «Sci-Fi + Ghibli», CloudGhibli shader, мягкие градиенты | Текстуры, gradient borders, soft shadows в UI |
| `defaultExecutionOrder=-200`, `[DefaultExecutionOrder]` для windows | Чёткий **слой анимаций и переходов** как отдельный модуль |

**Одна фраза:** мы построили правильную систему окон, но не построили **слоя времени и состояния**, который превращает окно в часть мира.

## 1. Что делает меню «immersive» в эталонных играх

Чтобы анализ был предметным, я разобрал 5 референсов и выделил **повторяющиеся приёмы**, а не «нам нравится NMS». Это не референс-лист — это **таксономия**.

### 1.1 Приёмы (что общего у NMS, BotW, Hades, Death Stranding, Destiny)

| # | Приём | NMS | BotW | Hades | Death Stranding | Destiny |
|---|---|---|---|---|---|---|
| 1 | **Параллаксный/глубинный фон** (UI реагирует на 3D-сцену за ним) | ✅ | ⚪ | ⚪ | ✅ | ⚪ |
| 2 | **Незаметный вход-выход** (fade+slide, не pop-in) | ✅ | ✅ | ✅ | ✅ | ✅ |
| 3 | **Шрифт и типографика как часть эстетики** (кастомный шрифт, кернинг, leading) | ✅ | ✅ | ✅ | ✅ | ✅ |
| 4 | **Анимация ЛЮБОГО изменения** (числа «тикают», ползунки плавно, вкладки сдвигаются) | ✅ | ⚪ | ✅ | ✅ | ✅ |
| 5 | **Звуковая сетка UI** (open/close/hover/select/error — разные звуки) | ✅ | ✅ | ✅ | ✅ | ✅ |
| 6 | **Мир продолжает жить за меню** (видишь движение облаков, других NPC, погоду) | ✅ | ✅ | ⚪ | ✅ | ✅ |
| 7 | **Игровые объекты как UI** (инвентарь = физические предметы на полке, а не иконки) | ⚪ | ✅ | ✅ | ✅ | ⚪ |
| 8 | **Привязка UI к состоянию игрока/мира** (тон меняется в бою/не в бою, днём/ночью) | ✅ | ✅ | ⚪ | ✅ | ✅ |
| 9 | **«Карта сама себя раскрывает»** (открытие = исследование, не загрузка данных) | ✅ | ✅ | ✅ | ⚪ | ✅ |
| 10 | **Иерархия фокуса** (один «главный» элемент, остальное — фоном) | ✅ | ✅ | ✅ | ✅ | ✅ |
| 11 | **Сетка как искусство** (композиция, не таблица) | ✅ | ⚪ | ✅ | ✅ | ✅ |

**Из 11 приёмов в проекте сейчас реализованы 0 из 11** (частично — №4 в DialogWindow typewriter, №6 — мир не блокируется во время меню, но меню и не реагирует на мир).

### 1.2 Что НЕ делают эти игры (и почему это важно для нас)

Чтобы «immersive» не превратилось в «over-designed», фиксирую и антипаттерны:

- ❌ **Не делают cinematic overlay с парящими элементами поверх игры без контекста** (типичная ошибка AAA — меню как «мёртвый» parallax где ничего не происходит).
- ❌ **Не используют skeuomorphism ради skeuomorphism** (инвентарь NMS — это **виртуальная полка**, не фотореалистичный шкаф).
- ❌ **Не делают анимации длиннее 0.4s на UI-события** (Hades использует snappy 0.15-0.2s; Destiny — то же).
- ❌ **Не блокируют мир за меню** (везде камера продолжает жить; BotW вообще показывает игровой мир сзади).

## 2. Анатомия immersive-меню — 5 слоёв

Я разделил «что нужно» на **5 слоёв**, не на фичи. Это позволяет проектировать изменения инкрементально, не ломая 10+ существующих окон.

```
┌────────────────────────────────────────────────────────────┐
│  Layer 5: World Response                                    │
│  UI реагирует на состояние мира (день/ночь, погода, бой)    │
│  и на состояние игрока (здоровье, фокус)                    │
├────────────────────────────────────────────────────────────┤
│  Layer 4: Audio Design                                      │
│  Каждое UX-событие имеет звук. Микрофон отключён во время   │
│  меню. Эмбиент за UI продолжается.                          │
├────────────────────────────────────────────────────────────┤
│  Layer 3: Composition & Layout                              │
│  Сетка как искусство. Иерархия фокуса. Кастомная            │
│  типографика. Soft borders, gradient fills.                 │
├────────────────────────────────────────────────────────────┤
│  Layer 2: Motion                                            │
│  Каждое изменение — анимировано. In/out transitions.        │
│  Parallax / depth. Snappy timing (≤0.25s).                  │
├────────────────────────────────────────────────────────────┤
│  Layer 1: Living Background                                 │
│  За UI виден живой мир (3D-сцена, эмбиент, частицы).        │
│  Параллакс. UI — «окно в мир», не «окно в Excel».           │
└────────────────────────────────────────────────────────────┘
```

### 2.1 Состояние слоёв в проекте (честная оценка)

| Слой | Состояние | Что есть / чего нет |
|---|---|---|
| **L1 Living Background** | ⚪ Частично | InventoryUI — прозрачный, за ним видно мир. Остальные окна — opaque `rgba(20,25,35,0.95)`. Параллакса нет нигде. |
| **L2 Motion** | ❌ Нет | USS `transition-property: width` (1 случай), `schedule.Execute(50ms)` (костыль repaint, не анимация). DOTween/LeanTween/Anime в `Packages/manifest.json` нет. |
| **L3 Composition** | ⚪ Частично | UITheme централизован, но layout везде «symmetric center panel» (одинаковая структура: panel-header + content + actions). Сетки как искусства нет. |
| **L4 Audio** | ❌ Нет | `UIManager.OpenSound/CloseSound/ClickSound/ErrorSound` — поля AudioClip, но НИ ОДНОГО вызова `PlaySound(ClickSound)` в существующем коде (нужно проверить в InventoryUI/TradeUI — `grep` покажет). |
| **L5 World Response** | ❌ Нет | UI не подписан на `DayNightCycle`, `WeatherSystem`, `PlayerState`. Открыл меню в полдень или в полночь — выглядит одинаково. |

## 3. Ключевые архитектурные решения (то, что нужно принять ДО кода)

Это решения, которые нельзя «принять по ходу» — они определяют, сколько кода придётся переписать.

### 3.1 Решение A: Что делать с уже работающими окнами?

**Контекст:** В проекте 10+ окон в разной степени готовности (CharacterWindow 1345 LOC, QuestTracker 230 LOC, InventoryWheel, TradeUI, MarketWindow, EscMenu, Keybindings, SkillBinding, SkillTree, Customisation, Crafting, DialogWindow, CommPanel, ShipCargoConsole). Все они используют один и тот же шаблон (см. `UI_TOOLKIT_GUIDE.md` §3 — канонический шаблон).

**Три стратегии (нужен твой выбор):**

| Стратегия | Что делаем | Плюсы | Минусы |
|---|---|---|---|
| **A1. Полный рефактор** | Каждое окно переписываем под новый стандарт | Консистентно | 2-3 спринта; ломаем всё на каждой итерации |
| **A2. Инкрементальный** | Новые окна (или 1-2 «витринных») — по новому стандарту; старые — мигрируем по одному | Безопасно, непрерывная играбельность | 1-2 окна будут «старыми» долго |
| **A3. Слой поверх** | Создаём **отдельный модуль** (например `UIFeel`/`MotionDirector`), который **оборачивает** существующие окна и добавляет анимации/звуки без переписывания | Быстрый эффект, не трогаем 10+ окон | Ограничен в выразительности, не решает L3 (composition) |

**Моя рекомендация (если хочешь услышать):** **A2 + A3 гибрид.** Сначала A3 даёт «слой движения и звука» (это можно сделать за 1 спринт, не трогая окна), потом A2 переписывает окна по одному с правильной композицией.

### 3.2 Решение B: Motion library — что использовать?

**Контекст:** Сейчас нет ни DOTween, ни LeanTween, ни собственного Tween'инга. UI Toolkit имеет встроенные transitions (CSS-подобные), но они слабые (только свойства, нет sequence/easing callbacks).

**Три варианта:**

| Вариант | Размер | Возможности | Стоимость для нас |
|---|---|---|---|
| **B1. Только UI Toolkit USS transitions** | 0 LOC | Fade, color, translate, scale. Ограниченный easing. Нет sequence/timeline. | Хватит на 70% случаев. Для остального — coroutine на `schedule.Execute()`. |
| **B2. DOTween** | +~80kb dll | Sequence, easing library, callbacks, можно твинить что угодно | Нужно добавить в `Packages/manifest.json` (см. AGENTS.md — это требует твоего approval) |
| **B3. Свой микро-Tweener** | ~300 LOC | Минимальный: tween(value, target, duration, easing, onUpdate, onComplete) | Полный контроль, не зависим от пакета |

**Моя рекомендация:** **B1 для старта** (бесплатно, не трогаем manifest), **B3 для продвинутых кейсов** (sequence, easing curves, custom transitions). B2 — отложить, если не упрёмся в лимит.

### 3.3 Решение C: Где живёт «motion design»?

UI Toolkit устроен так, что анимации можно делать в трёх местах:

1. **USS transitions** — для hover, focus, открытие-закрытие (L2 Motion, базовый).
2. **C# coroutines на `schedule.Execute()`** — для sequence, custom timing, callback.
3. **Внешний motion director** — отдельный компонент `UIFeel` / `MotionDirector`, который **слушает** события окон (Open/Close/TabChanged) и **оркестрирует** анимации и звуки.

**Рекомендация:** **Все три уровня**, но для разных задач:
- USS — для **per-element** (hover/focus).
- `schedule.Execute` — для **per-window** open/close.
- `MotionDirector` — для **cross-window** согласованности (например, когда открываешь SkillBinding поверх CharacterWindow — CharacterWindow должен «отступить»).

### 3.4 Решение D: Как UI связан с миром (L5)?

Это самое «глубокое» решение. Текущий UI — **detached**: `UIManager` ничего не знает про `DayNightCycle`, погоду, состояние корабля. Чтобы UI «откликался на мир», нужен один из двух подходов:

| Подход | Описание | Стоимость |
|---|---|---|
| **D1. UI poll** | UI каждый кадр проверяет состояние мира и подстраивается | Дёшево, но реактивно (лагает, шумит) |
| **D2. Event bus** | Мир публикует события (`DayChanged`, `WeatherChanged`, `CombatStarted`); UI подписан | Дороже в настройке, но правильно реактивно |

**Рекомендация:** **D2.** В проекте уже есть `GameEventBus` (проверить — если нет, создать; если есть — расширить). Без event bus любой L5 будет костылём.

## 6. Чего **не нужно** делать (защита от over-engineering)

Чтобы рефактор не превратился в бесконечную переделку:

| Не делать | Почему |
|---|---|
| ❌ Не пытаться сделать все 11 приёмов из §1.1 сразу | 80% «immersive» ощущения дают L1 (Living Background) + L4 (Audio) + L2 (Motion). L3/L5 — это уже «wow», но требуют огромной работы. |
| ❌ Не вводить новый UI framework (UI Builder, неткод UI, etc.) | UI Toolkit — правильный выбор. Проблема не в фреймворке, а в том, как мы его используем. |
| ❌ Не пытаться сделать UI «полностью 3D» (все элементы в мировом пространстве) | Это работает для инвентаря-полки, но **убивает читаемость** для табличных данных (торговля, контракты). Гибрид: да, не «всё 3D». |
| ❌ Не блокировать мир за меню | В проекте это и так не делается ✅ (cursor lock правильно ставится). НО: меню должно **использовать** это — показывать движение облаков, погоду, других игроков. |
| ❌ Не пытаться сделать «один motion на всё» | Разные окна требуют разного motion. EscMenu — fade-in (быстро, 0.15s). CharacterWindow — slide+scale (0.3s). DialogWindow — typewriter (40 chars/sec). |
| ❌ Не трогать `UIManager.OpenPanel/ClosePanel` API | Сигнатуры менять нельзя — иначе все 10+ окон нужно переписывать. Добавляем, не ломаем. |
