## HorizonVeil Analysis (2026-05-04)

### Task
Analyze why we don't use raymarch (despite discussing it), determine how to implement "завеса" (curtain) — cloud-filled bottom of horizon.

### CLOUDENGINE Analysis Results

**Files analyzed:**
- `C:\CLOUDPROJECT\CLOUDENGINE\shaders\cloud_advanced.frag` — full volumetric raymarch
- `C:\CLOUDPROJECT\CLOUDENGINE\unity_migration\Shaders_Root\VeilShader.shader` — flat plane veil (NOT volumetric)

**Key findings:**

| File | Approach | GPU Cost | Suitable? |
|------|----------|----------|-----------|
| cloud_advanced.frag | Volumetric raymarch (64 steps, 6 oct FBM) | ~3ms | Yes, but expensive |
| VeilShader.shader | Flat plane + noise + depth fade | ~0.2ms | **NO** — not volumetric |

**VeilShader is NOT volumetric** — just smoke on a flat plane. Cannot create "клубящаяся завеса со своими впадинами каньонами" (boiling curtain with valleys/canyons).

### Why Raymarch Was Rejected (Reconsidered)

Original rejection: "Full raymarch 64-128 steps = 4-20ms GPU — TOO EXPENSIVE"

But CLOUDENGINE shows:
- 64 steps = ~3ms (reasonable alone)
- 16 steps = ~1ms
- 8 steps = ~0.5ms

**Decision:** Use simplified raymarch (8-16 steps) + half-resolution = ~1-1.5ms total.

### HorizonVeil Design

**Approach:** Half-resolution volumetric raymarch for horizon curtain

| Parameter | Value | Reason |
|-----------|-------|--------|
| Resolution | 512x288 (0.5x) | 4x speedup, upscale acceptable |
| Raymarch steps | 12 | Balance quality/cost |
| FBM octaves | 2 | Fast noise |
| Layer altitude | 1000-3000m | Curtain at horizon |
| GPU estimate | ~1-1.5ms | With half-res |

**Integration:**
- HorizonVeilRenderer.cs in CloudManager
- VeilRaymarch.shader (based on CLOUDENGINE cloud_advanced.frag)
- Wind affects via shader uniform (_NoiseSpeed)
- Phase 4a (before Storm Authority)
