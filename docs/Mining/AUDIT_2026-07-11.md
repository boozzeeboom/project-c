## 3. Findings — по severity

### 🔴 CRITICAL: Dual XP path — WorldEventBus vs direct call

**Проблема:** GatheringServer НЕ публикует `MiningCompletedEvent` через `WorldEventBus`. Вместо этого — прямой вызов `StatsServer.Instance.ApplyXp()`:

```csharp
// GatheringServer.cs:164-174
// SESSION 1 refactor: прямой вызов StatsServer (не через WorldEventBus) — надёжнее, нет race-conditions.
try {
    var ss = ProjectC.Stats.StatsServer.Instance;
    if (ss != null) {
        var statType = ss.GetStatFor(ProjectC.Stats.XpSource.Mining);
        ss.ApplyXp(clientId, statType, (float)result.Quantity * 1.0f, $"Mining ×{result.Quantity} {result.ItemName}");
    }
} catch ...
```

При этом:
- `MiningCompletedEvent` **определён** в `WorldEvent.cs:186` (с комментарием «Publisher: GatheringServer (T-G03) в success-ветке CompleteGather»)
- `StatsServer` **подписан** на `MiningCompletedEvent` (строка 116/126) — **это мёртвый код**, никогда не вызывается
- `MiningCompletedEvent` **нигде не публикуется** (`grep` подтверждает: 0 результатов)

**Для сравнения — Crafting делает правильно:**
```csharp
// CraftingWorld.cs:139
WorldEventBus.Publish(new CraftingCompletedEvent { PlayerId = ..., StationNetId = ..., ... });
```

**Последствия:**
1. StatsServer имеет dead code в `OnMiningCompleted` — подписка есть, вызова нет
2. Другие системы (квесты, ачивки, аудио, HUD-нотификации) **не могут** подписаться на завершение майнинга
3. Архитектурная несогласованность: Crafting → WorldEventBus, Mining → прямой вызов
4. Технически XP начисляется (прямой вызов работает), но архитектура нарушена

**Рекомендация:** Унифицировать с Crafting-паттерном:
1. В `GatheringServer.TickActiveGathers` → после `case Completed`: `WorldEventBus.Publish(new MiningCompletedEvent {...})`
2. Убрать прямой вызов `StatsServer.ApplyXp` из GatheringServer (StatsServer сам обработает через WorldEventBus)
3. Либо — убрать dead code из StatsServer (`OnMiningCompleted` + подписку), если решено отказаться от WorldEventBus для майнинга. Но тогда нужно явно задокументировать это решение.

### 🟡 MEDIUM: Отсутствует префаб ResourceNode

**Проблема:** `query_project_assets("ResourceNode t:Prefab")` → 0 результатов. План (`20_IMPLEMENTATION_PLAN.md` §Фаза 6) предполагал создание `ResourceNode_Default.prefab`. Ноды размещены как raw GameObjects прямо в `WorldScene_0_0.unity`.

**Последствия:**

- Нет единого источника правды для структуры нода
- При изменении структуры (новое поле/компонент) — нужно обновлять все инстансы во всех сценах

**Рекомендация:** Создать `Assets/_Project/Prefabs/ResourceNode_Default.prefab` с полной структурой (NetworkObject + ResourceNode + BoxCollider isTrigger + MeshRenderer) и перевести существующие инстансы на prefab instances.

### 🟡 MEDIUM: MiningCompletedEvent — orphaned event class

**Проблема:** `MiningCompletedEvent` (WorldEvent.cs:186) определён, документирован, но **никогда не публикуется**. StatsServer подписан, но handler никогда не вызывается.

**Последствия:**
- Dead code в StatsServer
- Нарушение паттерна «Publish/Subscribe» для майнинга
- Будущие системы, которые захотят подписаться на майнинг, не смогут этого сделать

**Рекомендация:** См. CRITICAL выше — унифицировать с Crafting паттерном.

### 🟡 MEDIUM: GatherType.Lambering — без тестового нода

**Проблема:** 
- Enum `GatherType.Lambering = 1` существует в `ResourceNodeConfig.cs`

**Рекомендация:** Создать `ResourceNode_Tree.asset` с `_gatherType = Lambering` и разместить тестовый нод в `WorldScene_0_0`.

## 8. Итоговая оценка

**Система работает, все core-функции реализованы.** Технических багов в рантайме нет. Основная проблема — архитектурная: **несогласованность с Crafting-паттерном** (WorldEventBus) и orphaned `MiningCompletedEvent`. Это не ломает игру сейчас, но создаст проблемы когда другие системы (квесты, ачивки, звук) захотят реагировать на майнинг.

**Готовность к расширению: 7/10** — система самодостаточна, но отсутствие префаба и расхождение документации с кодом замедлят будущие интеграции.
