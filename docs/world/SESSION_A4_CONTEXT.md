## 🔍 Анализ единиц измерения (КРИТИЧНО!)

### WorldLandscape_Design.md §2.1:
```
1 Unity unit = 1 метр
Масштаб расстояний: 1:2,000 от реальности
Масштаб высот: 1:100 от реальности
```

### PeakDataFiller (текущий):
```
worldPosition.X = scaled units (1:2000) → 0, 600, 1200...
worldPosition.Y = scaled units (1:100) → 88.48, 72.00...
baseRadius = ??? units
```

### ВОПРОС: baseRadius в каких units?

Если baseRadius = 250 и это "игровые units":
- 250 units = 250 метров (1 unit = 1 метр)
- Реальный радиус Эвереста ≈ 30 км = 30,000 метров
- **250 ≠ 30,000** → baseRadius НЕПРАВИЛЬНЫЙ!

**Правильный baseRadius:**
- Реальный радиус Эвереста ≈ 30 км = 30,000 метров
- В игровых units (1 unit = 1 метр): 30,000 units
- Это ОГРОМНОЕ число.

**Вывод:** baseRadius должен быть в тех же units что и мир — **игровые метры**.

### Расчёт правильного baseRadius:

| Пик | Реальный радиус | Игровой baseRadius (units) |
|-----|----------------|---------------------------|
| Эверест | ~30 км | **30,000** |
| Монблан | ~15 км | **15,000** |
| Кибо | ~20 км | **20,000** |

**НО:** 30,000 units — это слишком много для Unity mesh!

### Решение: baseRadius в "scaled units"

Если baseRadius в scaled units (1:2000 от реальности):
- Эверест: 30 км / 2000 = **15 units**
- Монблан: 15 км / 2000 = **7.5 units**

Тогда meshHeight:
- Эверест: 15 * 1.8 = **27 units** (при h/r = 1.8)
- Это МАЛО! Гора 27 units высотой — это 27 метров в игре.

**Вывод:** НЕЛЬЗЯ использовать одну систему units для XZ и другую для baseRadius/meshHeight.

## 💡 РЕКОМЕНДУЕМЫЙ ПОДХОД (A4)

### Проблема фундаментальная:
- XZ координаты: 3000-8700 units между массивами
- Но горы должны быть 15-30 км шириной = 15,000-30,000 units
- **Горы ШИРЕ чем расстояния между ними!**

Это значит **горы НЕ могут быть в тех же units что и расстояния**.

### Решение: ОТДЕЛЬНЫЙ масштаб для гор

```
МИР (позиции): X,Z,Y в scaled units (1:2000, 1:100)
ГОРЫ (меши): baseRadius и meshHeight в "горных units"
             где 1 горная unit = 20 игровых units (эмпирически)
```

**ИЛИ проще:**

Пользователь нашёл scale = 60. Примем это как **константу**:

```csharp
// В MountainMeshBuilder
const float VISUAL_SCALE = 60f;

float meshHeight = baseRadius * hRatio * VISUAL_SCALE;
```

**НО:** тогда baseRadius = 250 → meshHeight = 27,000 → это слишком много для Unity mesh.

### Альтернативное решение (ЛУЧШЕЕ):

**Использовать конусы напрямую с правильными размерами:**

```csharp
// Эверест:
// - Должен быть ВИЗУАЛЬНО 500-800 units высотой
// - Должен быть ВИЗУАЛЬНО 300-500 units шириной
// - h/r = 1.5-2.0

meshHeight = 600f;  // Фиксированная высота для Эвереста
baseRadius = 350f;  // Фиксированный радиус
h/r = 1.71
```

**Таблица фиксированных размеров:**

| Пик | meshHeight | baseRadius | h/r |
|-----|-----------|-----------|-----|
| Эверест | 600 | 350 | 1.71 |
| Лхоцзе | 500 | 280 | 1.79 |
| Монблан | 400 | 220 | 1.82 |
| Кибо | 450 | 300 | 1.50 |
| Аконкагуа | 520 | 320 | 1.63 |
| Денали | 500 | 280 | 1.79 |

## 📝 Заметки по CapsuleCollider

**Проблема:** CapsuleCollider — это "капсула" (цилиндр + 2 полусферы). Для конуса горы это НЕ подходит.

**Варианты:**
1. **MeshCollider** — точная форма, но дороже (performance)
2. **Несколько CapsuleCollider** — аппроксимация конуса 2-3 капсулами
3. **Один большой CapsuleCollider** — грубая аппроксимация

**Рекомендация для A4:** Использовать **MeshCollider** с `convex = true` для коллизий корабля.
