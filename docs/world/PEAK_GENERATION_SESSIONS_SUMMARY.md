## 🎯 Выводы

### Корневая проблема:
**Фундаментальное несоответствие между системами координат:**
- X,Z = scaled units (1:2000 от реальности) — расстояния между пиками 3000-8700 units
- Y = scaled units (1:100 от реальности) — высоты пиков 88.48, 48.08, etc.
- baseRadius, meshHeight = непонятно в каких units

**Результат:** Горы либо в 60 раз меньше нужного, либо "пипки", либо непропорциональные иглы.

### Почему 5 итераций провалились:
1. **A3.1:** `meshHeight = worldPosition.y * 15` → h/r = 5.3 → ИГЛЫ
2. **A3.2:** Уменьшен baseRadius → h/r = 5.3 → ВСЁ РАВНО ИГЛЫ
3. **A3.3:** `meshHeight = baseRadius * hRatio` → правильные пропорции, но АБСОЛЮТНЫЙ РАЗМЕР МАЛЕНЬКИЙ
4. **A4:** Power-Law Cone + Shoulder + Noise → формы лучше, но материалы V2 не работают, размеры всё ещё неверные
5. **Пользователь:** Scale 40-60 на родителе Mountains → хоть как-то похоже на правду, но костыль

### Что нужно для решения (в будущем):
1. **Единая система единиц** — X,Y,Z в ОДНИХ units
2. **Явные размеры пиков** — НЕ формулы, а фиксированные значения:
   - Эверест: высота 800 units, радиус 400 units
   - Монблан: высота 650 units, радиус 350 units
   - и т.д.
3. **Правильные горные формы** — НЕ конусы/цилиндры, а что-то более сложное
4. **MeshCollider вместо CapsuleCollider**
5. **Возможно пребакед меши** — НЕ runtime generation, а запечённые в редакторе

## 🔜 Следующие шаги (когда вернёмся к задаче)

### Вариант A: Pre-baked meshes (Editor-time)
- Запекать меши в Editor → сохранять как .asset
- Runtime: просто instantiate prefab
- Плюсы: полный контроль над формой, нет проблем с генерацией
- Минусы: негибко, большие файлы

### Вариант B: Heightmap-based terrain
- Использовать Unity Terrain system
- Heightmap из реальных данных (SRTM, ASTER)
- Плюсы: реальная география, встроенный LOD
- Минусы: не подходит для isolated peaks, ограниченный контроль

### Вариант C: Ручное моделирование (Blender → Unity)
- Создать меши в Blender → импортировать
- Плюсы: полный контроль, красиво
- Минусы: долго, 29 пиков × 3 LOD = 87 мешей

### Вариант D: Hybrid approach
- Runtime generation для прототипа (упрощённые конусы)
- Pre-baked meshes для релиза
- Плюсы: быстро сейчас, красиво потом
- Минусы: двойная работа
