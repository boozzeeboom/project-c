#  Пошаговая разработка Project C

**Принцип:** "Медленнее = Быстрее"  
**Последнее обновление:** 2 апреля 2026 г.

---

## 🎯 Главный принцип

> **Не ломать рабочую версию!**

Вместо того чтобы делать большие изменения и рисковать рабочей версией, мы двигаемся **маленькими уверенными шагами**.

---

## 📋 Правила разработки

### 1. Один шаг = одна маленькая функция

❌ **Плохо:**
```
"Добавить систему облаков" (10 файлов, 2000 строк кода)
```

✅ **Хорошо:**
```
Шаг 1: Создать CloudLayerConfig.cs (ScriptableObject конфигурация)
Шаг 2: Создать CloudLayer.cs (менеджер одного слоя)
Шаг 3: Создать CloudSystem.cs (главный менеджер)
Шаг 4: Создать материалы для облаков
Шаг 5: Настроить сцену
```

---

### 2. Тестировать после каждого шага

После каждого шага:

```bash
# 1. Проверить компиляцию
# В Unity: подождать пока исчезнут ошибки

# 2. Проверить что работает
# В Unity: нажать Play, проверить функцию

# 3. Закоммитить если работает
git add .
git commit -m "Шаг 1: CloudLayerConfig.cs создан"
git push
```

---

### 3. Резервная копия перед серией шагов

Перед началом серии изменений:

```bash
# Создать тег
git tag backup/pre-cloud-system
git push upstream --tags

# Или создать ветку
git checkout -b feature/cloud-system
```

---

### 4. Откат если шаг не работает

Если после шага что-то сломалось:

```bash
# Отменить изменения
git restore .

# Или откатиться к последнему коммиту
git reset --hard HEAD
```

## 📝 Чек-лист перед коммитом

- [ ] Код компилируется без ошибок
- [ ] Игра запускается
- [ ] Новая функция работает
- [ ] Старые функции не сломались
- [ ] Нет временных файлов (`.disable`, `.bak`)
- [ ] `.meta` файлы на месте

---

## 🎯 Критерии маленького шага

**Хороший шаг:**
- ✅ 1 файл или 1 класс
- ✅ 50-200 строк кода
- ✅ Можно протестировать за 1 минуту
- ✅ Можно откатить за 30 секунд
- ✅ Понятное описание коммита

**Плохой шаг:**
- ❌ 5-10 файлов
- ❌ 1000+ строк кода
- ❌ Тестирование 10+ минут
- ❌ Сложно откатить
- ❌ "Добавил фичу" в коммите
