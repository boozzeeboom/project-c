# ARTIFACT DEEP ANALYSIS: FloatingOriginMP + NGO NetworkTransform Conflict\n\n**Date:** 17.04.2026, 20:00\n**Project:** ProjectC_client\n**Status:** ANALYSIS COMPLETE - ROOT CAUSES IDENTIFIED\n\n---\n\n## EXECUTIVE SUMMARY\n\nThe multiplayer position sync artifacts are caused by a fundamental architectural conflict between:\n1. **FloatingOriginMP** - world origin shifting system\n2. **NGO NetworkTransform** - Unity Netcode for GameObjects position synchronization\n\nThese two systems operate independently, without coordination, causing the observed jitter, teleportation, and desync artifacts in host-client multiplayer.\n\n---\n\n## ROOT CAUSE ANALYSIS\n\n### Problem 1: Two Independent Position Worlds\n\n**Server/Host Side:**\n- Uses CharacterController.Move() for local player movement\n- NetworkTransform sends position to clients at ~15Hz\n- Position is **interpolated** on receiving clients\n\n**Client Side:**\n- Receives NetworkTransform updates\n- Interpolates to target position using Lerp\n- On pause, interpolation stops = shows real server position\n\n**The Conflict:**\n- Host broadcasts: Player moved to (100, 5, 200)\n- Client receives: NetworkTransform says (100, 5, 200)\n- Client interpolates: lerping toward (100, 5, 200)\n- FloatingOriginMP shifts world: WorldRoot -= (100000, 0, 0)\n- NetworkTransform correction fires: snap to server position!\n- Result: Jitter, teleport, or rubber-band\n\n---\n\n## WHY COOLDOWN APPROACH DOES NOT WORK\n\nThe cooldown approach tries to ignore the problem rather than solve it:\n\n1. **Symptom treatment, not cause fix**\n   - Cooldown prevents correction for 1 second\n   - But next frame, another shift can occur\n\n2. **No synchronization between systems**\n   - FloatingOriginMP does not communicate shift timing to NetworkTransform\n   - NetworkTransform does not know world shifted\n\n3. **Interpolation continues during cooldown**\n   - Client position keeps moving toward server position\n   - Server position is in old coordinate space\n\n4. **Race condition**\n   - World shifts in LateUpdate\n   - NetworkTransform correction fires in FixedUpdate or next LateUpdate\n   - Timing is not deterministic\n\n---\n\n## ALTERNATIVE SOLUTIONS\n\n### Solution 1: Disable NetworkTransform Client-Side Correction (RECOMMENDED)\n\n**In Inspector on Player prefab NetworkTransform:**\n- Interpolate Position: 0\n- Interpolate Rotation: 0\n- Interpolate Scale: 0\n\n**Expected result:**\n- Less jitter during world shifts\n- But may have teleport steps instead of smooth motion\n\n### Solution 2: Use NetworkRigidbody\n\n**PROS:**\n- Server-authoritative by default\n- Less prone to client-side artifacts\n- Syncs velocity, not position directly\n\n**CONS:**\n- Requires physics-based movement (Rigidbody)\n- CharacterController is not physics-based\n- Would need significant refactoring\n\n### Solution 3: Coordinate Space Normalization\n\nAll systems work in a normalized coordinate space.\n\n### Solution 4: Manual Server-Authoritative Sync (MOST RELIABLE)\n\nReplace NetworkTransform with custom RPC-based sync.\n\n---\n\n## CONCRETE RECOMMENDATIONS\n\n### Priority 1: Immediate Fix\n\n1.
