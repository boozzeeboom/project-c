## 🎯 Главный принцип

> **Не ломать рабочую версию!**

Вместо того чтобы делать большие изменения и рисковать рабочей версией, мы двигаемся **маленькими уверенными шагами**.

---

## 📋 Правила разработки

### ⚠️ НЕ ТРОГАТЬ `.meta` ФАЙЛЫ

> **КРИТИЧНО: Никогда не создавать и не редактировать `.meta` файлы вручную!**

Unity **автоматически** создаёт `.meta` файлы для каждого ассета в проекте. Эти файлы содержат:
- **GUID** — уникальный идентификатор ассета (ссылки в префабах, сценах, скриптах)
- **Настройки импорта** — как Unity обрабатывает файл
- **Связи с другими ассетами**

**Последствия ручного изменения:**
- ❌ Сломанные ссылки в префабах и сценах
- ❌ Ошибки компиляции и `TypeLoadException`
- ❌ Потерянные материалы, текстуры, скрипты
- ❌ Необходимость пересоздавать ассеты заново

**Правило:**
- ✅ Unity создаёт `.meta` автоматически при импорте
- ✅ Не удалять `.meta` файлы (даже если удаляешь сам ассет — Unity разберётся)
- ✅ Не создавать `.meta` файлы вручную
- ✅ При коммите — `.meta` файлы коммитятся вместе с ассетами

**Если случайно создал `.meta`:**
```bash
# Удалить файл и дать Unity пересоздать
rm path/to/file.meta
# Или в Windows:
del "path\to\file.meta"
```

### 1. Один шаг = одна маленькая функция

❌ **Плохо:**
```
"Добавить систему облаков" (10 файлов, 2000 строк кода)
```

✅ **Хорошо:**
```
Шаг 1: Создать CloudLayerConfig.cs (ScriptableObject конфигурация)
Шаг 2: Создать CloudLayer.cs (менеджер одного слоя)
Шаг 3: Создать CloudSystem.cs (главный менеджер)
Шаг 4: Создать материалы для облаков
Шаг 5: Настроить сцену
```

---

### 2. Тестировать после каждого шага

После каждого шага:

```bash
# 1. Проверить компиляцию
# В Unity: подождать пока исчезнут ошибки

# 2. Проверить что работает
# В Unity: нажать Play, проверить функцию

# 3. Закоммитить если работает
git add .
git commit -m "Шаг 1: CloudLayerConfig.cs создан"
git push
```

---

### 3. Резервная копия перед серией шагов

Перед началом серии изменений:

```bash
# Создать тег
git tag backup/pre-cloud-system
git push upstream --tags

# Или создать ветку
git checkout -b feature/cloud-system
```

---

### 4. Откат если шаг не работает

Если после шага что-то сломалось:

```bash
# Отменить изменения
git restore .

# Или откатиться к последнему коммиту
git reset --hard HEAD
```

## 🎯 Критерии маленького шага

**Хороший шаг:**
- ✅ 1 файл или 1 класс
- ✅ 50-200 строк кода
- ✅ Можно протестировать за 1 минуту
- ✅ Можно откатить за 30 секунд
- ✅ Понятное описание коммита

**Плохой шаг:**
- ❌ 5-10 файлов
- ❌ 1000+ строк кода
- ❌ Тестирование 10+ минут
- ❌ Сложно откатить
- ❌ "Добавил фичу" в коммите
